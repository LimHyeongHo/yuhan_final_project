<font color="#0056b3">### 1. 키페어 생성 (RSA KeyPair)</font>
  키페어 생성은 pki 모듈에서 수행되는 것이 맞습니다.
  PkiTestUtilController.java에서 RSA 알고리즘(KeyPairGenerator.getInstance("RSA"))을 사용하여 공개키(Public Key)와 개인키(Private Key) 쌍을
  생성합니다.
   * 공개키: 서버(User 엔티티)에 저장되어 이후 검증에 사용됩니다.
   * 개인키: 사용자 기기(Client)에 안전하게 저장되어 서명 시에 사용됩니다.
   *  1. 개인키 전달 방식 (API 응답)
        generateKeys() 메서드를 보면, 서버에서 RSA 키 쌍(Public/Private)을 생성한 뒤 JSON 형태의 API 응답으로 두 키를 모두 클라이언트에 전달합니다.
       - publicKey: 서버에 저장 (기기 바인딩 검증용)
       - privateKey: 클라이언트로 전송 (사용자 소유)

      2. 구체적인 저장 위치 (기기 내부)
        이 프로젝트의 구조상, 클라이언트(사용자 기기)가 이 개인키를 받으면 다음과 같은 안전한 저장소에 저장하게 됩니다. (플랫폼에 따라 다름)

        (1) 모바일 앱 (Android/iOS)
        가장 권장되는 방식이며, 실제 서비스에서 주로 사용하는 위치입니다:
         - Android: Android KeyStore System에 저장합니다. 이 저장소는 하드웨어 수준(TEE/SE)에서 보호되어, 루팅된 기기라도 키를 평문으로 추출하기 매우
           어렵습니다.
         - iOS: Keychain 서비스에 저장합니다. 앱이 삭제되어도 키를 유지하거나, 생체 인증(FaceID/TouchID)과 연동하여 보안을 강화할 수 있습니다.
      
        (2) 웹 브라우저 (JavaScript)
        웹 환경이라면 다음과 같은 곳에 저장됩니다:
         - Web Crypto API (IndexedDB): 브라우저의 안전한 데이터베이스에 키 객체 형태로 저장합니다. extractable: false 옵션을 주면 JavaScript 코드로도 키
           원본을 읽을 수 없게 보호할 수 있습니다.
         - Local Storage: (보안상 취약하지만 간단한 테스트용) 단순히 문자열 형태로 저장할 수도 있습니다. 하지만 이 방식은 XSS 공격에 취약하므로 실제 운영
           서비스에서는 피해야 합니다.
      
        3. 기기 바인딩의 핵심 원리
        이 프로젝트에서 개인키가 "안전하게 저장된다"는 말의 구체적인 의미는 다음과 같습니다:
      
         1. 키의 고립: 개인키는 한 번 기기에 저장되면 외부로 유출되지 않도록(Non-exportable) 관리됩니다.
         2. 서명 수행: 로그인이 필요할 때 서버는 챌린지 값을 보내고, 기기는 저장된 개인키로 서명만 해서 다시 보냅니다. 즉, 개인키 자체를 서버로 보내는
            것이 아니라, "내가 이 키를 가지고 있다"는 증명(서명값)만 보냅니다.
         3. 바인딩 완성: 만약 사용자가 다른 기기에서 접속하려 한다면, 그 기기에는 해당 개인키가 없기 때문에 서버의 공개키 검증을 통과할 수 없습니다.
            이것이 바로 RSA 알고리즘을 통한 기기 바인딩의 핵심입니다.
      
        요약
        질문하신 "어디에 저장하나?"의 답은:
         - 기술적 위치: Android KeyStore, iOS Keychain, 또는 브라우저의 Web Crypto API.
         - 보안적 특징: 키가 저장소 밖으로 나가지 않고, 오직 서명(Signing) 연산에만 사용되도록 설계되어 있습니다.

 <font color="#0056b3">### 2. 1인 1계정 시스템 (CI HMAC 관리)</font>
  1인 1계정 시스템은 말씀하신 cihmac 로직을 통해 관리됩니다.
  PkiService.java의 generateCiHash(String ci) 메서드를 보면:
   * 사용자의 고유 식별값인 CI(Connecting Information)를 입력받아,
   * 서버만 알고 있는 비밀키(serverSecret)와 함께 HMAC-SHA256 알고리즘을 사용하여 해시값을 생성합니다.
   * 이 해시값을 통해 동일한 사용자가 중복으로 가입하는 것을 방지하거나(1인 1계정), 개인정보 노출 없이 사용자를 식별하는 용도로 사용됩니다.

  <font color="#0056b3">### 3. 기기 바인딩 (RSA 알고리즘 검증)</font>
  기기 바인딩은 RSA 알고리즘을 통한 서명 검증(verifySignature)으로 걸러내는(Enforce) 방식이 맞습니다.
  PkiService.java의 validateChallenge 메서드 로직은 다음과 같습니다:
   1. 서버가 사용자에게 챌린지(Challenge) 값을 보냅니다.
   2. 사용자 기기는 자신의 개인키(RSA Private Key)로 이 챌린지에 서명하여 다시 서버로 보냅니다.
   3. 서버는 미리 등록된 사용자의 공개키(RSA Public Key)로 서명을 검증합니다.
   * 결과적으로: 해당 기기에 저장된 개인키가 등록된 공개키와 쌍을 이루지 않으면(즉, 등록되지 않은 기기라면) 인증에 실패하게 되므로, 자연스럽게 특정
     기기에서만 접근 가능하도록 바인딩되는 효과를 가집니다.

  ---

  요약하자면:
   - 키 생성: pki에서 RSA 키 쌍 생성.
   - 1인 1계정: HMAC-SHA256 기반의 CI Hash(cihmac)로 관리.
   - 기기 바인딩: 서버에 저장된 RSA 공개키와 기기에 저장된 개인키 간의 서명 검증을 통해 수행.
